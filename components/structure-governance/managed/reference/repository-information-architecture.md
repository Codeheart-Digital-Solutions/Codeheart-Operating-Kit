# Repository Information Architecture

Use this reference before introducing durable names, new folder boundaries, or externally visible
identifiers.

## Decision Order

1. Identify the owner and lifecycle of the concept.
2. Decide whether the concept is reusable kit doctrine or consumer-specific guidance.
3. Place durable generic rules in Operating Kit managed docs.
4. Place local repository or product rules in consumer-owned docs.
5. Keep externally visible names descriptive until adoption history justifies aliases.
6. Avoid duplicate durable guidance that can drift.

## Naming

Prefer clear names that identify the owning domain and behavior. Avoid short aliases in first
release surfaces unless they are required for compatibility.
